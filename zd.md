昨天早上好奇，为什么插着耳机的时候，读屏的无障碍音量有时候会从手机里窜出来，于是就趁它窜出来的时机，抓了一段 ADB 日志。

日志分析显示，国内很多大厂应用调用完安卓的某些音频 API 之后没有释放，或者存在乱用 API 的情况，导致系统的音频路由受到影响。具体来说，这些应用在播放语音消息等场景下调用了 `setSpeakerphoneOn(true)`，本意是从听筒切到扬声器外放，但用完之后没有释放，也没有判断用户是否插着耳机——插着耳机的时候音频本来就不走听筒，这个调用完全是多余的。残留的通信路由设置会影响系统对无障碍音频流的路由决策，而屏幕阅读器本来就是要顺着系统音频路由走的，于是就默默承受了这份伤害。这就造成，在微信听了语音、刷了视频号，在美团 APP 点击视频选项卡、甚至在首页什么都不动，在豆包结束视频通话，还有各种各样的场景之后，系统的音频路由就被污染了，无障碍播报会从耳机跳到扬声器外放。

这个 APP 所做的事情就是：你戴着耳机的情况下，一旦音频路由受到影响，它会通过系统回调的方式捕获路由的改变，然后调用公开 API 把通信设备设回耳机，让你戴着耳机的时候，无障碍播报继续从耳机里输出；不戴耳机的时候，应用就悄悄休眠，等你插入耳机后，再被系统回调唤醒并检测一次，如果音频路由没有及时切回去，就执行一次修复。用到的 API 都是 Android 12 以上的公开接口，不需要 root。逻辑其实并不复杂，我自己使用了一天，感觉效果还是可以的。

但现在的问题就是，我手里只有一台破红米，发给过几个朋友使用，不过大家大概率都是破红米，在破红米上稳定性挺好的，其他手机咱就不知道了。如果有人想要试试的话，我就发个测试包出来，看看效果到底怎么样，值不值得将它继续更新下去。毕竟它值得完善的地方还是很多的，比如说无障碍声道被扬声器抢走这件事，成因不止这一个；再比如说既然实现了对音频路由的调整，是不是可以增加更多的玩法，一切都还不知道。

我起初只是好奇这个问题，让 AI 写了一段 Java 来复现，然后想着把这个逻辑做成 APP 挂在后台，帮我修声道，没想到它的复杂性以及未来可能有的机遇会不小，那就发出来大家一起乐呵乐呵吧。

这个项目本身不难，没什么价值，我顺手开源了，估计也就是垃圾代码一类的。但是圈子里小人太多，安装包本体我做了签名校验，不要随便改俩字符串就发出来玩。虽说这个签名校验能绕过吧，但防君子不防小人，咱也没有办法。
